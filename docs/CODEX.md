# duplo-next 現行仕様書
## 1. アプリ概要

- アプリ種別: Next.js 14 App Router ベースの貸し借り・立替記録アプリ
- 主な用途: グループ内の支払い履歴を記録し、支払総額や差額を確認する
- 想定利用者:
  - 開発者本人
  - 家族、友人、身近なコミュニティなど少人数の利用者
- 運用方針:
  - 一般公開サービス向けの大規模運用は想定しない
  - 身内向けにシンプルに使えることを優先する
  - 必要最低限の運用負荷で継続利用できることを重視する
- 表示上の名称:
  - `<title>`: お金貸し借り清算アプリ
  - ヘッダー: 奢り奢られ清算
  - PWA マニフェスト名: 割り勘清算アプリ
- UI 技術:
  - React 18
  - Tailwind CSS
  - shadcn/ui
  - SWR
  - framer-motion
  - react-hook-form + zod
- バックエンド技術:
  - Next.js Route Handler
  - PostgreSQL 接続に `pg` を使用
  - `DATABASE_URL` を使って Neon 互換 DB に接続する想定

## 2. プロダクト方針

少人数で使う前提のため、今後の仕様判断は以下の優先順位を基準にします。

- 優先すること:
  - 身内が迷わず使えるシンプルな導線
  - 支払い記録の登録・修正・確認がすぐできること
  - 開発者が自分で保守しやすい構成
  - データを壊さないこと、復旧しやすいこと
- 優先度を下げること:
  - 不特定多数向けの onboarding
  - 大規模アクセスを前提にした設計
  - 複雑な権限管理や課金導線
  - マーケティング向けの LP や公開導線
- 最低限必要なこと:
  - 秘密情報をコードや公開場所に漏らさないこと
  - 身内以外が簡単に触れないようにすること
  - バックアップや手動修正ができること

## 3. 画面構成

### 3.1 共通レイアウト

- 全画面で共通ヘッダーを表示
- 最大幅は `max-w-4xl`
- `UpdateNotifier` により Service Worker の更新検知 UI を右下に表示可能
- `manifest.json` と PWA アイコンを `<head>` に設定

### 3.2 `/` トップページ

ホーム画面として `HomePage` を表示します。

- 入力項目:
  - `pageName`
  - `description`
  - `hostname`
  - `username`
- バリデーション:
  - 4 項目とも `z.string()` のみ
  - 必須チェックや文字数制限は未設定
- 初期値:
  - `a`, `b`, `c`, `d`
- 送信時の実装:
  - `/api/createPage` に `POST`
  - 送信 payload はフォーム入力を使わず、常に `{ username: "John", password: "a" }`
- ボタン押下時の遷移:
  - `onClick` で固定パス `"/lpdelpdlpelpkopelpd"` に遷移
- 現状の実質的な役割:
  - ページ作成フォームの見た目はあるが、ページ作成処理は未完成
  - API 応答に応じた遷移制御も未実装

### 3.3 `/:pageId` 貸し借り詳細ページ

`LoanPage` を表示し、`pageId` をキーに履歴を取得します。

- 上部に「支払額」セクションを表示
- 下部に「履歴」セクションを表示
- 右下のフローティングボタンから `/:pageId/form` へ遷移

#### 支払額セクション

`TotalLoan` で同一 API データを集計し、タブ表示します。

- タブ:
  - `総額`
  - `差額`
- 総額タブ:
  - `payer_name` ごとに `pay_amount` を合算して表示
- 差額タブ:
  - 基準値を `平均` または `最大` から選択
  - 表示値:
    - 平均選択時: `平均支払額 - 各人の総支払額`
    - 最大選択時: `最大支払額 - 各人の総支払額`
- ローディング中:
  - スケルトン UI を表示

#### 履歴セクション

- データ取得先: `/api/getHistory/:pageId`
- 初期表示順:
  - 日付の降順
- ソート切替:
  - 昇順/降順を切替可能
- 検索:
  - 検索アイコンで検索バーを開閉
  - `note` に対する部分一致検索
  - 大文字小文字は区別しない
- 月見出し:
  - `pay_time` を `YYYY年M月` に変換して月単位の見出しを挿入
- データなし:
  - `履歴がありません` を表示
- ローディング中:
  - スケルトン UI を表示

### 3.4 `/:pageId/form` 支払い登録ページ

新規履歴登録用の `FormPage` を表示します。

- 画面上部:
  - 戻るボタン
  - タイトル `支払い登録`
- 入力項目:
  - `payerName`: 1 文字以上必須
  - `amount`: 数値に変換し、1 以上必須
  - `payTime`: `Date`
  - `note`: 1 文字以上必須
- 初期値:
  - `payerName = "山田"`
  - `amount = 1000`
  - `payTime = new Date()`
  - `note = "メモ"`
- 日付入力:
  - Popover + Calendar で単日選択
- 送信時:
  - `/api/postHistory` に `POST`
  - payload には `eventId: 1` を固定で付与
  - `pageId` から `eventId` への変換はしていない
- 成功時:
  - HTTP 200 のとき `/:pageId` に戻る

## 4. 履歴カードの仕様

履歴一覧の各行は `History` コンポーネントで表示します。

- 表示内容:
  - 支払い者名
  - 金額
  - メモ
  - 日付
- 編集:
  - ペンアイコンからダイアログを開く
  - 入力項目とバリデーションは新規登録画面と同等
  - 保存時に `/api/updateHistory` へ `POST`
  - 成功時は SWR の `mutate` で一覧を再取得
  - 500 エラー時は `alert("エラーが発生しました")`
- 削除:
  - ゴミ箱アイコンから確認ダイアログを開く
  - `/api/deleteHistory` に `POST`
  - 成功時は SWR の `mutate` で一覧を再取得

## 5. API 仕様

### 5.1 `POST /api/createPage`

- リクエスト JSON を受け取って `console.log` する
- DB 書き込みなし
- 常に `{ message: "success" }` または `{ message: "error" }` を返す
- ページ作成機能としては未完成

### 5.2 `GET /api/getHistory/:eventId`

- 実際には `eventId` という名前だが、検索条件には `treat.url = :eventId` を使う
- SQL:
  - `treat` と `history` を `event_id` で JOIN
  - 指定 URL に紐づく履歴一覧を返す
- 返却カラム:
  - `t.url`
  - `t.last_edit`
  - `h.history_id`
  - `h.payer_name`
  - `h.pay_amount`
  - `h.pay_time`
  - `h.note`
- エラー時:
  - `{ message: "error" }`

### 5.3 `POST /api/postHistory`

- 受信項目:
  - `eventId`
  - `payerName`
  - `amount`
  - `payTime`
  - `note`
- SQL:
  - `history(event_id, payer_name, pay_amount, pay_time, note)` に INSERT
- 正常時:
  - HTTP 200
  - `{ message: "success" }`
- 異常時:
  - HTTP 500
  - `{ message: error }`

### 5.4 `POST /api/updateHistory`

- 受信項目:
  - `historyId`
  - `payerName`
  - `amount`
  - `payTime`
  - `note`
- SQL:
  - `history_id` 指定で UPDATE
- 正常時:
  - HTTP 200
  - `{ message: "success" }`
- 異常時:
  - HTTP 500
  - `{ message: error }`

### 5.5 `POST /api/deleteHistory`

- 受信項目:
  - `historyId`
- SQL:
  - `history` から対象 ID を DELETE
- 正常時:
  - HTTP 200
  - `{ message: "success" }`
- 異常時:
  - HTTP 500
  - `{ message: error }`

## 6. データベース前提

コードから読み取れる前提は以下です。

### 6.1 `history` テーブル

少なくとも以下のカラムを前提にしています。

- `history_id`
- `event_id`
- `payer_name`
- `pay_amount`
- `pay_time`
- `note`

### 6.2 `treat` テーブル

少なくとも以下のカラムを前提にしています。

- `event_id`
- `url`
- `last_edit`

### 6.3 接続方式

- `src/app/api/Pool.ts` で `pg.Pool` を生成
- 接続文字列は `process.env.DATABASE_URL`
- `ssl: true`

## 7. PWA / 更新通知

- `public/sw.js` を Service Worker として登録
- `install` 時に `skipWaiting()` を実行
- `activate` 時に既存キャッシュを全削除
- `fetch` 時は `caches.match` を試し、なければ通常の `fetch`
- `UpdateNotifier` は `updatefound` を検知すると画面右下に更新通知を表示
- 更新ボタン押下時:
  - `SKIP_WAITING` メッセージを送信
  - その後 `window.location.reload()`

## 8. ログ

- 開発環境:
  - `pino-pretty` で整形出力
  - ログレベルは `debug`
- 本番環境:
  - `logs/app.log` に出力
  - ログレベルは `info`
- ただし現状の API 実装では `logger` はほぼ未使用で、`console.log` / `console.error` が中心

## 9. 少人数運用前提での設計メモ

現時点では実装済みではないものの、このプロジェクト方針で特に重要になる観点を整理します。

- 認証:
  - 本格的な会員登録機能は必須ではない
  - 共有 URL、固定メンバー向けの簡易認証、合言葉方式などでも運用可能
- データ運用:
  - 間違った履歴をあとで直せることが重要
  - 管理者である開発者が DB を直接確認・修正しやすいことが望ましい
- セキュリティ:
  - 公開しない前提でも `DATABASE_URL` などの秘密情報管理は必須
  - リポジトリや公開 URL に機密情報を載せない
- UI:
  - 身内が直感的に使えるラベル・導線を優先する
  - 汎用的な説明文より、実際の利用シーンに合う日本語文言を優先する
- 機能追加の優先順位:
  - まずは「作成」「記録」「修正」「削除」「閲覧」が安定して動くこと
  - その後に共有方法や見た目を整える

## 10. 今後追加したい機能

少人数での実運用を踏まえ、次の機能追加を想定する。

### 10.1 計上に含めない機能

履歴を残しつつ、集計対象から除外できる機能を追加したい。

- 想定用途:
  - 入力ミスとして一旦無効化したい
  - 参考情報として履歴は残したいが、清算額には反映したくない
  - 立替記録ではなく、メモ用途の支払いを残したい
- 想定挙動:
  - 各履歴に「計上対象 / 計上対象外」の状態を持てる
  - 履歴一覧には表示する
  - `総額` と `差額` の計算からは除外する
  - 編集画面から切り替えられる形が望ましい
- 実装メモ:
  - `history` テーブルに `is_included` のような真偽値カラムを追加する案が有力
  - 既存データは `true` 扱いにするのが自然
  - API と集計ロジックの両方でこのフラグを考慮する必要がある

### 10.2 あとからまとめて計上対象に戻す機能

個別の履歴登録時にはいったん「計上に含めない」として保存し、あとから必要な履歴だけまとめて計上対象に戻せる機能を追加したい。

- 想定用途:
  - イベント中は仮の記録として残し、あとで確定した履歴だけ集計に反映したい
  - 清算や確認が終わったタイミングで、複数の履歴をまとめて有効化したい
  - 一時的に除外していた履歴を、あとでまとめて戻したい
- 想定挙動:
  - `計上対象外` の履歴だけを一覧表示できる
  - その中から複数件を選択できる
  - 実行後、選択した履歴が `計上対象` に変わる
  - 総額と差額の計算に即時反映される
- UI 案:
  - `/:pageId/form` とは別に「まとめて計上」画面を作る
  - `計上対象外` の履歴だけをチェックボックス付きで並べる
  - 選択した履歴を一括で `計上対象` に更新する
- 実装メモ:
  - `history` テーブルの `is_included` を `false -> true` に一括更新する
  - 対象は現在の `pageId` に紐づく履歴に限定する
  - 新しい履歴を追加するのではなく、既存履歴の状態変更として扱う

## 11. 現状の制約・未実装事項

コード上、以下は未完成または注意が必要です。

- トップページの「作成」は実際にはページを新規作成していない
- トップページのフォーム値は API に渡っていない
- 作成後の遷移先 URL は固定文字列で、API レスポンスとも連動していない
- 新規履歴登録は `eventId: 1` 固定で登録するため、URL 上の `pageId` と保存先イベントが一致しない可能性がある
- `getHistory` は URL 文字列で検索する一方、`postHistory` は数値風の `eventId` を受ける構造で、識別子の設計が揃っていない
- 一部 UI 文言が仮置きのまま
  - 例: ホームの各ラベルがすべて `Page Name`
  - 例: 説明文 `This is your public display name.`
- エラー表示は限定的で、主に `alert` または無表示
- `src/constants.ts` は空ファイル
- `env.local` に接続情報が置かれているため、公開運用しない場合でも取り扱いに注意が必要

## 12. 実装ベースの利用フロー

現状コードから見た利用フローは次の通りです。

1. `/` を開く
2. 「貸し借り管理を作成する」を押す
3. 固定 URL `/:pageId` 形式に見えるページへ遷移する
4. `/:pageId` で履歴一覧と集計を見る
5. 追加ボタンから `/:pageId/form` へ進み支払いを登録する
6. 履歴一覧で編集・削除する

トップページのページ作成機能は未完成のため、実際には既存の `pageId` を直接開く利用が前提に近い実装です。

## 13. 当面の改善優先順位

少人数向けアプリとして実用化するなら、優先度は以下の順が妥当です。

1. トップページの作成機能を実際に動くようにする
2. `pageId` と `eventId` の対応関係を整理し、履歴登録先を正しくする
3. ホーム画面の文言や入力項目を実態に合わせて直す
4. エラー表示を分かりやすくする
5. 「計上に含めない」フラグを履歴に持たせる
6. 「まとめて計上」機能を追加する
7. 身内向けの共有方法を決める
